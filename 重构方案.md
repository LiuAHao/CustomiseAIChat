# Go Agent Platform 重构方案

## 1. 项目定位

### 1.1 目标

将当前仓库从“单 AI 聊天系统原型”重构为一个基于 Go 的多 Agent 管理平台。

平台首版不是要一次性做成完整分布式系统，而是先交付一个可运行、可扩展、可演进的模块化单体，覆盖以下核心目标：

- 多工作区、多 Agent 管理
- Agent 版本化与发布
- Tool / MCP / Skill / RAG 的统一接入位
- 任务执行、步骤追踪、审批与审计
- 前后端分离的控制台与事件推送

### 1.2 当前原则

本次重构遵循以下原则：

1. 全新开始，不兼容旧 C++ 数据模型
2. 先固定架构边界，再逐步接入真实基础设施
3. 先做平台主链路，再做高级能力
4. 先模块化单体，再决定是否拆服务

## 3. 架构设计

### 3.1 总体架构

首版采用模块化单体：

- 一个 API 进程承载 HTTP / WebSocket / 控制面入口
- 一个 worker 进程承载异步执行和调度消费
- 业务逻辑统一沉淀在 `internal/app`
- 领域对象统一沉淀在 `internal/domain`
- 基础设施能力统一沉淀在 `internal/platform`

这意味着当前不是微服务架构，但已经为后续拆分留下边界。

### 3.2 分层边界

#### `cmd`

进程入口层，只负责启动，不承载业务规则。

- `cmd/api`：API 服务入口
- `cmd/worker`：Worker 服务入口

#### `internal/app`

应用层，负责组织业务流程。

这里定义“系统做什么”，例如：

- 登录
- 创建工作区
- 创建 Agent
- 发布版本
- 发起任务
- 审批恢复
- 调度触发

#### `internal/domain`

领域层，负责定义系统讨论的核心对象。

当前核心对象包括：

- `auth`
- `workspace`
- `agent`
- `tool`
- `session`
- `task`
- `approval`
- `audit`

#### `internal/platform`

基础设施层，负责提供运行能力。

当前已有：

- `memory`
- `events`
- `llm`

后续会增加：

- `postgres`
- `redis`
- `queue`
- `rag`
- `mcp`

#### `internal/transport`

传输层，负责对外协议。

当前已有：

- `http`
- `ws`

后续可增加：

- `grpc`
- `admin`
- `webhook`

## 6. 技术选型策略

### 6.1 当前策略

当前阶段不应一次性锁死所有基础设施，应分“接口先行”和“实现后补”两层推进。

### 6.2 首版推荐选型

| 类别 | 当前建议 | 说明 |
|------|---------|------|
| HTTP | 标准库 `net/http` | 当前已实现，足够支撑首版 |
| WebSocket | 当前自实现轻量版本 | 先满足事件流，后续可替换 |
| 配置 | 自定义配置加载 | 当前已足够 |
| 日志 | 标准日志起步 | 后续可替换成 Zap |
| 存储 | 内存实现 + PostgreSQL schema 预留 | 先跑通主链路 |
| LLM | mock provider | 先验证编排与任务链路 |

### 6.3 第二阶段推荐选型

| 类别 | 推荐选型 | 说明 |
|------|---------|------|
| 主数据库 | PostgreSQL | 比 MySQL 更适合 JSON、审计、任务状态 |
| 缓存 | Redis | 用于会话热点、锁、幂等和调度辅助 |
| 队列 | 可延后，优先轻量实现 | 不建议首版立即接 RabbitMQ |
| 向量库 | Qdrant 或 pgvector | 等 RAG 进入实施阶段再定 |
| LLM 接入 | OpenAI / DeepSeek / Ollama | 基于统一接口接入 |
| 观测 | Zap + OpenTelemetry | 在主链路稳定后接入 |

### 6.4 为什么不建议过早上 RabbitMQ

当前项目规模下，首版更重要的是：

- 明确任务模型
- 明确状态迁移
- 明确幂等键和错误处理

如果这些没收敛，先接 MQ 只会把调试复杂度放大。

建议顺序是：

1. 先实现进程内 / 数据库驱动任务执行
2. 再把执行抽到 worker
3. 最后再考虑正式队列

## 10. 风险与控制

### 风险 1：过早引入太多基础设施

问题：

- 增加调试成本
- 打断主链路验证

策略：

- 接口先行
- 实现分阶段接入

### 风险 2：`internal/app` 继续膨胀

问题：

- 业务集中导致难维护

策略：

- 当某块逻辑稳定后，拆成独立应用服务

### 风险 3：前端过早追求完整后台

问题：

- 首页壳子还没完全稳定，过早铺页面会返工

策略：

- 先把信息架构和关键操作流定住，再扩展多页面

### 风险 4：RAG / MCP / Skill 并行推进

问题：

- 平台边界容易混乱

策略：

- 先 Tool，再 MCP，再 Skill Registry
- RAG 独立成“知识增强能力线”

---

## 11. 当前结论

当前项目已经不是“从零开始”，而是已经完成了第一阶段的平台骨架。

现在最合理的推进顺序不是继续扩展页面，也不是立刻引入全部基础设施，而是：

1. 先把持久化做实
2. 再把任务 / 审批 / 事件主链路稳定化
3. 然后接真实模型
4. 再上 RAG
5. 最后补 MCP / Skill / 队列 / 完整观测

这条路线最符合当前仓库状态，也最不容易返工。
